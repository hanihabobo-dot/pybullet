53. REPLACE PUSH WITH PICK-AND-PLACE FOR OCCLUDER RELOCATION
================================================================================
Where: pddl/domain_pddlstream.pddl, pddl/stream.pddl, streams.py,
       pddlstream_planner.py, test_full_pipeline.py

What:  Occluders are currently relocated via a push action that teleports
       the object (p.resetBasePositionAndOrientation). The robot arm is
       never involved. The push stream invents fake destinations. Execution
       ignores all plan parameters and re-derives its own push direction.
       This entire pipeline is broken (#50, #51) and unfixable without
       rewriting it from scratch.

       The domain already has PICK and PLACE as two separate actions:

         pick(?o, ?b, ?g, ?q)  — grasp object ?o at boxel ?b
         place(?o, ?b, ?g, ?q) — release object ?o at boxel ?b

       These two actions, combined with move, can relocate an occluder:

         pick(occluder, occ_boxel, grasp, q_pick)
           → robot grasps the occluder and lifts it
         move(q_pick, q_place, free_boxel, traj)
           → robot carries the occluder to a free-space boxel
         place(occluder, free_boxel, grasp, q_place)
           → robot places the occluder in the free-space boxel

       This is deterministic (no sliding uncertainty), uses existing robot
       motion infrastructure (move_robot_smooth + gripper), and the planner
       already knows how to sequence pick → move → place.

       But currently only target objects have the (Obj) predicate, so the
       planner cannot pick/place occluders.

Fix:   Two parts: enable occluders for pick/place, then disable push.

  PART 1 — Make occluders graspable (planner changes):
       (a) pddlstream_planner.py _build_init(): for each OBJECT boxel,
           add (Obj occluder_id) so the pick and place actions apply.
           Also add (obj_at_boxel_KIF occluder_id occluder_boxel) — we
           can see the occluder, so we know it's there. The existing
           (obj_at_boxel occluder_id occluder_boxel) is already emitted.
       (b) No PDDL domain changes needed — pick and place already work
           for any (Obj ?o). The planner will generate pick → move → place
           plans for occluders the same way it does for targets.
       (c) No stream changes needed — sample_grasp and compute_kin_solution
           already accept any object ID. They will produce grasps and IK
           solutions for occluders automatically.
       (d) The planner's derived predicates handle the rest: once the
           occluder is placed in a free-space boxel (where no
           blocks_view_at fact exists), view_clear becomes true, and
           the sense action becomes available for the shadow behind it.

  PART 2 — Comment out push (code cleanup):
       (a) domain_pddlstream.pddl: comment out the push action. Keep the
           text for reference but make the planner unable to use it.
       (b) stream.pddl: comment out the sample-push-config stream.
       (c) pddlstream_planner.py _get_stream_map(): remove the
           'sample-push-config' entry.
       (d) streams.py: comment out sample_push_config() and its helper
           methods (the push-specific IK logic). Keep sample_grasp,
           plan_motion, compute_kin_solution.
       (e) test_full_pipeline.py: remove the push execution branch
           (the entire 'if action_name == "push"' block, ~lines 280-320).
           Remove compute_push_displacement() entirely.
       (f) pddlstream_planner.py _build_init(): remove (occluder_at)
           emissions — that predicate was only needed for the push stream
           domain binding. Remove push-related predicates if unused.

  PART 3 — Add place execution (new code):
       test_full_pipeline.py needs a place execution handler alongside the
       existing pick handler. Pick and place are separate actions:

       execute_pick (already exists, see #7 for fixing it):
         - Move robot to kin_solution config
         - Open gripper
         - Lower to grasp position
         - Close gripper
         - Attach object via p.createConstraint
         - Lift

       execute_place (new):
         - Move robot to kin_solution config (arm above the destination)
         - Lower to placement height
         - Open gripper
         - Release p.createConstraint
         - Lift arm clear

       The plan dispatcher needs a new 'elif action_name == "place"' branch
       that calls execute_place with the plan's (obj, boxel, grasp, config)
       parameters.

  Expected plan shape after this fix:
       1. pick(occluder_1, occ_boxel, grasp_1, q_pick)
       2. move(q_pick, q_place, free_boxel_3, traj_1)
       3. place(occluder_1, free_boxel_3, grasp_1, q_place)
       4. sense(target_1, shadow_1)
          → if found:
       5. move(q_place, q_pick_target, shadow_1, traj_2)
       6. pick(target_1, shadow_1, grasp_2, q_pick_target)
          → if not found: replan for next shadow

52. MOVE EXECUTION IGNORES PLAN CONFIGS — USES HARDCODED OFFSETS
================================================================================
Where: test_full_pipeline.py move execution (~lines 336-352)
What:  When the plan says move(q1, q2, boxel, traj), execution throws
       away q2 and uses hardcoded type-dependent offsets:
         [0, -0.3, 0.2] for shadow boxels
         [0,  0,   0.15] for object/other boxels
       Then re-runs IK from scratch via move_robot_to_pos(). The IK-solved
       config q2 from the plan is dead data.

Fix:   Call move_robot_smooth(robot_id, q2.joint_positions) to drive the
       arm to the plan's IK-solved configuration. Remove all hardcoded
       offsets and move_robot_to_pos() calls. After this, compute_ik()
       and move_robot_to_pos() in robot_utils.py become dead code (see
       #31 note in archive).

7. PICK — HARDCODED OFFSETS, IGNORES PLAN GRASP
================================================================================
Where: test_full_pipeline.py execute_pick() (~lines 510-540)
What:  execute_pick() uses hardcoded offsets from target_pos:
       - +0.15 Z (move above)
       - +0.05 Z (lower to grasp)
       - +0.3 Z (lift)
       The grasp and kin_solution parameters from the plan are completely
       ignored. IK is recomputed from scratch via move_robot_to_pos().
       Object attachment uses p.createConstraint (not physics-based grasping).

       Same pattern as #52: the plan carries real geometric data (grasp
       pose from sample_grasp, IK-solved config from compute_kin_solution)
       but execution throws it away and uses hardcoded offsets.

       This is now MORE CRITICAL because pick is used for both targets
       AND occluders (after #53). Every occluder relocation and every
       target pickup goes through execute_pick().

Fix:   (1) Move robot to the plan's kin_solution config (the IK-solved
       joint angles for the grasp approach) using move_robot_smooth().
       (2) Use the grasp pose from the plan to compute the approach,
       contact, and lift waypoints relative to the object's actual
       position — not hardcoded Z offsets.
       (3) The constraint-based attachment (p.createConstraint) is
       acceptable for simulation — physics-based grasping with friction
       is a separate improvement.
       (4) Once this is done, compute_ik() and move_robot_to_pos() in
       robot_utils.py become fully dead code (see #31 note in archive).

44. SHADOWS NOT RECOMPUTED AFTER OCCLUDER RELOCATION — STALE REGISTRY
================================================================================
Where: test_full_pipeline.py, boxel_data.py
What:  After relocating an occluder (via pick-and-place), the BoxelRegistry
       is NOT updated. The shadow boxels (their AABBs, centers, extents)
       still reflect the pre-relocation geometry. The sensing function
       (sense_shadow_raycasting) uses the shadow boxel's AABB to determine
       where to cast rays — but that AABB is for the old shadow, not the
       actual post-relocation geometry.

       With pick-and-place this is LESS CRITICAL than with push: since the
       occluder is physically removed (picked up and placed elsewhere),
       rays through the old shadow AABB will pass through empty space and
       can detect the target. The stale AABB just means rays may cover a
       slightly wrong area, but the target is detected by body ID, so any
       ray that hits it works.

Fix:   After relocating, either:
       (a) Recompute shadows for the relocated occluder only (incremental).
       (b) Re-run the full boxel computation (expensive but correct).
       (c) Accept the stale AABB as good enough (target detection by body
           ID compensates). Document this as a known simplification.

5. MOTION PLANNING — LINEAR INTERPOLATION, NO COLLISION CHECKING
================================================================================
Where: streams.py lines 413-432
What:  Motion "planning" is linear interpolation in joint space between two
       configs. No collision detection, no obstacle avoidance, no RRT/PRM.
       Comment on line 413 says "TODO: Implement actual motion planning".

Fix:   Implement a real motion planner (e.g., RRT-Connect or PRM using
       PyBullet's collision checking). The motion planner should NOT reason
       about line of sight — it should only produce collision-free paths.
       Line-of-sight clearing is the responsibility of pick-and-place
       occluder relocation, not motion planning.
