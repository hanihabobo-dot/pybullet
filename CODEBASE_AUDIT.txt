5. MOTION PLANNING — LINEAR INTERPOLATION, NO COLLISION CHECKING
================================================================================
Where: streams.py lines 413-432
What:  Motion "planning" is linear interpolation in joint space between two configs.
       No collision detection, no obstacle avoidance, no RRT/PRM.
       Comment on line 413 says "TODO: Implement actual motion planning".

Fix:   Implement a real motion planner (e.g., RRT-Connect or PRM using PyBullet's
       collision checking). The motion planner should NOT reason about line of
       sight — it should only produce collision-free paths. Line-of-sight clearing
       is the responsibility of the push action and its streams.

20. BoxelStreams CLASS IS NEVER USED BY THE PLANNER
================================================================================
Where: streams.py (BoxelStreams class, ~490 lines), pddlstream_planner.py
What:  pddlstream_planner.py defines its OWN stub generators that produce
       string names (_gen_push_config, _gen_sensing_config, etc.).
       The real BoxelStreams class in streams.py (with actual IK via
       PyBullet, proper viewpoint sampling, grasp generation) is NEVER
       connected to the planner.

       This is the ROOT CAUSE of issue #4 — streams produce strings
       BECAUSE the planner uses its own stubs instead of BoxelStreams.

Fix:   Wire BoxelStreams generators into pddlstream_planner.py's
       _get_stream_map(). Replace the stub generators with calls to
       the BoxelStreams methods. This is the single most impactful fix
       for making the planner geometry-aware.

26. heuristic_ik() — MEANINGLESS FAKE IK FALLBACK  [DONE]
================================================================================
Where: streams.py lines 268-285
What:  When robot_id is None (no PyBullet robot loaded),
       _compute_sensing_ik() falls back to _heuristic_ik() which
       produces "plausible-looking" joint configs by:
         offset = (ee_pos - [0.5, 0, 0.5]) * 0.5
         perturb home config by offset * 0.3

       These are NOT inverse kinematics — they are random perturbations
       of the home pose. The resulting configurations have no relationship
       to the requested end-effector position. They exist only to let
       tests run without a robot, but they could silently corrupt
       results if used when a robot IS expected.

Fixed: (1) Added is_heuristic flag to RobotConfig (default False). Configs
       from _heuristic_ik() are marked is_heuristic=True so downstream
       code can detect and reject them when real IK is expected.
       (2) BoxelStreams.__init__ logs a WARNING when robot_id is None,
       alerting early that all IK will use the heuristic fallback.
       (3) _heuristic_ik() logs a WARNING on every call with the
       requested ee_pos, making it impossible to miss silently.
       (4) Docstring rewritten to honestly describe what the function
       does (arbitrary perturbation, NOT inverse kinematics).

32. PDDL DOMAIN MODELS SOLUTION, NOT ROBOT CAPABILITIES  [REOPENED]
================================================================================
Where: pddl/domain_pddlstream.pddl, test_full_pipeline.py, stream.pddl

Previously done (cosmetic):
       Renamed push_aside → push, sense_shadow → sense. Replaced
       scenario-specific predicates (occluder_blocking, occluder_aside,
       casts_shadow) with spatial predicates (blocks_view, view_clear).
       These were naming improvements, not structural fixes.

What's STILL wrong:

  (A) push(?obj ?region ?q) is "push to clear a view", not a capability.
       The action does NOT track where the object moves. It has no
       ?b_from or ?b_to. Its effects only flip view predicates:
         (not (blocks_view ?obj ?region))
         (view_clear ?region)
       The object's position is never updated — no (obj_at_boxel ?obj ...)
       change. Push is "clear the camera's line of sight," not "move an
       object from here to there." That is the scenario encoded as an action.

  (B) Execution embeds sensing inside push. verify_push_cleared_view()
       casts rays from the camera after every push to check if the view
       was cleared — this is a sensing action that the planner never
       planned. It hardcodes that every push must be followed by a
       view check. The planner should decide what to verify and when.

  (C) Occluder positions are not tracked. The init state has no
       (obj_at_boxel ?occ ?b) for occluders, so the planner cannot
       reason about where occluders are or where they will end up.

Fix:   Make push a SPATIAL CAPABILITY — "move object from A to B."

       No other open issues need to be solved first. #20 (wiring
       BoxelStreams) is independent — the domain structure can change
       while generators still produce string placeholders. #5, #6,
       #7, #44 are all independent.

       Break into three phases to avoid regressions:

  PHASE 1 — Track occluder positions (additive, nothing breaks):
       Add (obj_at_boxel ?occ ?b) for every occluder in the init
       state. Currently only the target's position is tracked. No
       action uses this yet, so nothing changes in behavior.

  PHASE 2 — Rewrite push as spatial + derived predicates:

       Push action requirements:
         - Parameters: the object, where it is now, where it should
           go, the robot start config, end config, and the push
           trajectory. Pushing is a MOTION — the robot moves while
           contacting the object.
         - Preconditions: object must be at source, robot at start
           config, valid push trajectory exists, hand empty.
         - Effects: PURELY SPATIAL. Object moves from source to
           destination. Robot config updates from start to end.
           No view predicates.

       Derived predicates (PDDLStream/Fast Downward support these):
         - blocks_view(?obj, ?region): derived from object positions.
           True when ?obj is at a location that geometrically blocks
           the camera's view to ?region.
         - view_clear(?region): derived as the negation — true when
           no object blocks the view to ?region.
         Push does NOT mention views at all. The planner automatically
         computes view state from spatial positions.

       Push stream requirements:
         - Input: the object and its current location.
         - Output: destination, robot start config, end config,
           and push trajectory.
         - The stream computes WHERE and HOW to push based on
           geometry. Destination is an output, not hardcoded.

       Execution:
         - Push moves the object from source to destination.
         - Keep verify_push_cleared_view() as a safety net for now.

  PHASE 3 — Move verification from push to sense:
       Remove verify_push_cleared_view() from push execution. Sense
       needs a third outcome: "view still blocked" (rays hit the
       occluder, not the target and not empty space). If blocked,
       belief state must NOT mark the shadow as "not_here" — it
       hasn't been observed yet. This prevents a failed push from
       corrupting the belief state.

36. sense_shadow IGNORES ROBOT POSITION — USES FIXED CAMERA  [DONE]
================================================================================
Where: test_full_pipeline.py lines 324-338
What:  The PDDL plan moves the robot to a sensing config before sensing:
         1. move(q, q_sense, shadow_boxel, traj)
         2. sense_shadow(obj, shadow, occ, q_sense)

       But sense_shadow_raycasting() casts rays from the FIXED camera
       position (env.camera_position), not from the robot's end-effector
       or any config-dependent viewpoint. The robot's actual position has
       zero effect on sensing.

Fixed: Removed the sensing_config stream and robot-config dependency from
       the sense action entirely. We use a fixed scene camera, so the plan
       should not pretend the robot needs to be positioned for sensing.
       (1) domain_pddlstream.pddl: sense action now takes (?o ?region)
           instead of (?o ?region ?q). Removed (at_config ?q),
           (sensing_config ?region ?q) preconditions. Removed
           sensing_config predicate.
       (2) stream.pddl: removed sample-sensing-config stream.
       (3) pddlstream_planner.py: removed _gen_sensing_config generator
           and its stream-map entry.
       (4) test_full_pipeline.py: sense dispatch unpacks 2 params
           instead of 3.
       (5) streams.py: removed sample_sensing_config() and
           test_visibility() (both were dead code for the fixed-camera
           architecture). Updated module docstring.
       The planner will no longer generate move-to-sense steps. Plans
       become: push → sense → move-to-pick → pick (was: push →
       move-to-sense → sense → move-to-pick → pick).

41. NO VERIFICATION THAT PUSH CLEARED LINE OF SIGHT  [REOPENED — see #32]
================================================================================
Where: test_full_pipeline.py lines 259-288
What:  After pushing an occluder (teleportation + 30 physics steps), the
       code proceeds to the next action without verifying that camera
       rays can now penetrate the shadow region. If the push displacement
       was insufficient, or physics settling moved the occluder back
       partially, the subsequent sense_shadow will fail to detect the
       target even if it IS there.

Previously done: Added verify_push_cleared_view() which casts rays from
       the camera after push + physics settling. If shadow is still
       blocked, the occluder is not marked as moved and replanning is
       triggered.

Still wrong: verify_push_cleared_view() is sensing logic embedded in
       push execution. The planner never planned a view-check — the
       push executor hardcodes "after every push, do sensing." This
       bypasses the planner and models the solution in execution.

       The correct architecture is described in #32's fix: push
       execution just moves the object. Sense execution detects
       whether the view is blocked and reports the appropriate
       outcome. verify_push_cleared_view() should be removed from
       push and its logic absorbed into sense (which already casts
       rays from the fixed camera).

44. SHADOWS NOT RECOMPUTED AFTER PUSH — STALE REGISTRY
================================================================================
Where: test_full_pipeline.py lines 259-288, boxel_data.py
What:  After pushing an occluder, the BoxelRegistry is NOT updated. The
       shadow boxels (their AABBs, centers, extents) still reflect the
       pre-push geometry. The sensing function (sense_shadow_raycasting)
       uses the shadow boxel's AABB to determine where to cast rays —
       but that AABB is for the pre-push shadow, not the actual post-push
       geometry.

Fix:   After pushing, either:
       (a) Recompute shadows for the pushed occluder only (incremental).
       (b) Re-run the full boxel computation (expensive but correct).
       (c) At minimum, use the target's actual PyBullet position for
           ray-casting rather than the stale shadow AABB.
       Option (c) is partially in place — the target is detected by body
       ID, so it works if any ray happens to hit it. The stale AABB just
       means rays may not cover the right area.
