44. SHADOWS NOT RECOMPUTED AFTER OCCLUDER RELOCATION — STALE REGISTRY
================================================================================
Where: test_full_pipeline.py, boxel_data.py
What:  After relocating an occluder (via pick-and-place), the BoxelRegistry
       is NOT updated. The shadow boxels (their AABBs, centers, extents)
       still reflect the pre-relocation geometry. The sensing function
       (sense_shadow_raycasting) uses the shadow boxel's AABB to determine
       where to cast rays — but that AABB is for the old shadow, not the
       actual post-relocation geometry.

       With pick-and-place this is LESS CRITICAL than with push: since the
       occluder is physically removed (picked up and placed elsewhere),
       rays through the old shadow AABB will pass through empty space and
       can detect the target. The stale AABB just means rays may cover a
       slightly wrong area, but the target is detected by body ID, so any
       ray that hits it works.

Fix:   After relocating, either:
       (a) Recompute shadows for the relocated occluder only (incremental).
       (b) Re-run the full boxel computation (expensive but correct).
       (c) Accept the stale AABB as good enough (target detection by body
           ID compensates). Document this as a known simplification.

59. OBJECTS FLY AWAY ON FIRST GRASP — CONSTRAINT IMPULSE / COLLISION DURING PICK
================================================================================
Where: test_full_pipeline.py execute_pick(), robot_utils.py close_gripper()

What:  On the first pick action, surrounding objects visibly fly away from
       the grasped object. The grasp itself also looks "supernatural" — the
       object snaps to the gripper rather than being smoothly acquired.

       Likely causes (need logging to confirm):
       (a) p.createConstraint() offset mismatch: the constraint uses
           grasp.position as parentFramePosition, but if the EE isn't
           at exactly obj_pos + grasp.position when the constraint is
           created, PyBullet applies a large corrective impulse in the
           next stepSimulation() to satisfy the constraint.  This impulse
           can propagate to adjacent objects via contact forces.
       (b) close_gripper() physically closes the fingers (30 simulation
           steps with force=50). If the fingers contact the object or
           adjacent objects before the constraint is created, the contact
           forces push them.  The gripper closes BEFORE createConstraint,
           so the object isn't welded yet and can be pushed.
       (c) The approach/contact motion (move_robot_smooth) may collide
           with neighboring objects in the cluttered scene.  Planning
           checks collisions with gripper tolerance enabled, but the
           actual motion applies physics forces that push things.
       (d) No simulation settling after createConstraint — the next call
           is move_robot_smooth(lift_joints) immediately, while the
           constraint solver is still producing corrective forces.

Debug: Add logging around execute_pick():
       - Log EE position vs expected contact_ee at constraint creation time
       - Log all object positions before and after each phase:
         before approach, after approach, after close_gripper,
         after createConstraint, after lift
       - Compute and log displacement of each nearby object to identify
         which phase causes the scatter
       - Log the constraint error (actual vs desired relative transform)

60. SIMULATION-WIDE FLICKERING DURING PLANNING — RENDERING TOGGLE ARTIFACTS
================================================================================
Where: robot_utils.py is_config_collision_free(), is_path_collision_free(),
       solve_ik(); streams.py _pybullet_ik()

What:  After the arm-flickering fix (wrapping resetJointState with
       configureDebugVisualizer COV_ENABLE_RENDERING 0/1), the arm no
       longer flickers, but the entire simulation now exhibits periodic
       visual glitches — the scene briefly looks "weird" every couple of
       seconds.

       Root cause: the rendering toggle is called thousands of times per
       planning cycle (once per is_path_collision_free call, once per
       solve_ik/IK call).  Each enable→disable→enable cycle may force
       an OpenGL buffer swap or partial frame render, producing a brief
       flash.  The fix addressed per-call arm teleportation but introduced
       a high-frequency global rendering toggle that has its own visual
       side effects.

Fix:   Instead of toggling rendering per-function-call, disable rendering
       ONCE at the start of the entire planning phase (before
       planner.plan()) and re-enable ONCE after planning completes.
       During planning the robot should appear frozen in its current pose.
       This eliminates thousands of toggle cycles per plan and prevents
       any rendering artifacts.  The per-function toggles in
       is_config_collision_free, solve_ik, _pybullet_ik can then be
       removed (or kept as fallback for standalone calls outside planning).

       Alternatively, use a reference-counted rendering lock: first
       disable increments a counter, enable decrements; rendering is
       only actually toggled when the counter transitions 0→1 or 1→0.

Debug: Log the number of COV_ENABLE_RENDERING toggle calls per planning
       cycle to quantify the overhead.  Compare visual behavior with
       rendering disabled for the full planning phase vs per-call toggle.

61. SENSE ACTION DEVIATES FROM PROPOSAL — NO CONDITIONAL EFFECTS  [CRITICAL]
================================================================================
Where: pddl/domain_pddlstream.pddl (sense action, lines 120-133),
       test_full_pipeline.py (sense execution, lines 303-337),
       proposal Section 4.4.2 (Listing 2)

What:  The PDDL sense action unconditionally adds (obj_at_boxel ?o ?region)
       and (obj_pose_known ?o).  It assumes the target will always be found
       ("optimistic" comment in the domain file).  It has NO observation
       stream, NO conditional effects, and NO ?obs_status parameter.

       The proposal explicitly defines sense_Boxel_for_object with:
         - stream_find_sensing_config (robot must move to a sensing pose)
         - stream_get_sensing_outcome (returns found / not_found)
         - (when (= ?obs_status found) ...) → sets obj_in_Boxel
         - (when (= ?obs_status not_found) ...) → sets obj_not_in_Boxel

       None of these exist.  The planner always produces plans that assume
       sensing succeeds on the first shadow it checks.  When execution
       reveals the target is NOT there, the Python code breaks out of the
       action loop and replans from scratch.

       This is a fundamental architectural deviation: the proposal's design
       lets the planner reason about both sensing outcomes in a single plan.
       The implementation uses purely reactive replanning — plan optimistically,
       execute, replan on failure.  While this works, it cannot produce
       multi-step search strategies and is strictly less capable than the
       proposed conditional design.

Impact: The planner cannot produce a plan like "sense shadow_A; if empty,
       sense shadow_B" because the PDDL model has no way to represent the
       "not found" branch.  Every plan assumes the first sense succeeds.

62. PDDL-EXECUTION CONFIG STATE DRIFT — pick/place DON'T UPDATE at_config
================================================================================
Where: pddl/domain_pddlstream.pddl (pick: lines 161-179, place: 185-203),
       test_full_pipeline.py execute_pick(), execute_place()

What:  The pick action requires (at_config ?q) in its precondition and
       does NOT delete or replace it in effects.  After pick, the PDDL
       state still says the robot is at the pick config ?q.  But physically
       the robot has moved through approach → contact → close → lift
       waypoints and is at the lift position.

       Same issue for place: PDDL says robot is at the place config,
       but physically the robot is at the retreat position.

       Within a single multi-action plan, the PDDL model and physical
       robot state diverge after every pick or place.  The next move
       action's precondition (at_config ?q1) checks against the PDDL
       state (the pick/place config), but the actual robot joint positions
       are at the lift/retreat config.  The trajectory that plan_motion
       computed was from ?q_pick to ?q_next, not from ?q_lift to ?q_next,
       so the first segment of the trajectory is a jump.

       The execution code compensates by updating current_config after
       each pick/place (line 354 and 373) and passing it to the next
       planner.plan() call.  But within a SINGLE plan that has multiple
       pick/place/move sequences, the drift accumulates.

Fix:   Either:
       (a) Add (not (at_config ?q)) to pick/place effects, plus a new
           "post_pick_config" or similar that is set by the action.
       (b) Accept this as a modeling simplification and document it.
           The reactive replanning loop means each plan rarely has more
           than one pick/place before replanning resets state.

63. HEURISTIC IK FALLBACK REACHABLE IN PRODUCTION — NO GUARD AT EXECUTION
================================================================================
Where: streams.py _heuristic_ik() (lines 311-335),
       streams.py BoxelStreams.__init__() (lines 126-130),
       pddlstream_planner.py test_planner() (line 395),
       test_full_pipeline.py (execution loop)

What:  _heuristic_ik() produces joint angles with NO kinematic relationship
       to the requested end-effector pose.  It perturbs the home config by
       an arbitrary function of the target position.  The resulting config
       is marked is_heuristic=True, and a warning is logged.

       HOWEVER, no code in the execution pipeline ever checks is_heuristic.
       If BoxelStreams is constructed without robot_id (which test_planner()
       does on line 395), the planner generates kinematically invalid
       configs that execute_pick/execute_place/move would blindly follow.

       In the normal test_full_pipeline.py flow, robot_id is always set,
       so _heuristic_ik is never reached.  But there is no assertion or
       guard preventing it.  A single misconfiguration (e.g., robot_id=None
       due to a setup error) would produce silent garbage behavior.

Fix:   Add an assertion at the top of plan() or at the beginning of
       execution: if any config in the plan has is_heuristic=True, abort
       with a clear error message.  Or remove the fallback entirely —
       if robot_id is None, refuse to plan.

64. SHADOW RAYCAST USES WRONG RAY ORIGINS — ray_starts VARIABLE UNUSED
================================================================================
Where: shadow_calculator.py, line 92 vs line 102

What:  Line 92 creates ray_starts = [cam_pos] * len(corners), which is
       the camera position repeated for each back corner — the correct
       origin for camera-through-corner shadow projection.

       Line 102 calls p.rayTestBatch(corners, ray_ends), using corners
       (the object's back corners) as ray origins instead of ray_starts.
       The variable ray_starts is computed but never passed to the API.

       The actual behavior: rays start from the object's back corners
       and go outward along the camera-to-corner direction.  This is
       a reasonable approximation for shadow extent — the rays trace
       the shadow boundary outward from the object.  But it is NOT the
       standard camera-to-surface projection described in the proposal.
       The ray_starts variable is dead computation.

Impact: The shadow geometry may be slightly different from what a camera-
       through-corner projection would produce, particularly for objects
       far from the camera or at extreme angles.  In practice, the
       approximation works for the current tabletop scene.

Fix:   Either use ray_starts (camera-through-corner projection) or
       document the current corner-outward method as intentional and
       remove the unused ray_starts variable.

67. _build_init() HARDCODES ASSUMPTION THAT TARGETS ARE ONLY IN SHADOWS
================================================================================
Where: pddlstream_planner.py _build_init(), lines 269-286

What:  For every OBJECT boxel (line 280-281) and every FREE_SPACE boxel
       (lines 285-286), the code adds:
         (obj_at_boxel_KIF target boxel_id)
       This means "we know the target is NOT in this boxel" — because
       obj_at_boxel is not also added.  Only shadow boxels are left as

       implementation precludes scenarios where a target is on an open
       part of the table but the robot doesn't know which one it is.

Impact: Low for the current hidden-object scenario (targets are always
       behind occluders by construction).  But this prevents extending
       the system to more general search tasks without reworking _build_init.

PROPOSAL ALIGNMENT GAPS
================================================================================
These are features described in the proposal that have no corresponding
implementation. They are not code bugs — they are missing functionality
that the thesis promises.

PA-1. No evaluation framework (proposal Section 5)
      The proposal defines metrics (success rate, planning time, plan cost,
      scalability) and three baselines (uniform voxelization, fixed semantic
      regions, POMDP-based TAMP). None exist in the codebase. There is no
      benchmarking code, no baseline implementations, no metric collection,
      no experiment runner.

PA-2. No recursive object discovery in free space (proposal Section 4.2)
      The proposal says: "If new objects are found within a partition, the
      process repeats: bound the biggest object and define its occluded
      space." The actual FreeSpaceGenerator does a single-pass octree that
      marks cells as FREE or OCCUPIED. It never detects new objects inside
      partitions — all objects are detected upfront by oracle_detect_objects.

PA-3. K-literal design differs from proposal (proposal Section 4.4.1)
      The proposal defines two predicates: obj_in_Boxel (K(p)) and
      obj_not_in_Boxel (K(¬p)), with "possibly in" being the absence of
      both. The implementation uses a single obj_at_boxel_KIF predicate:
      present = "we know", absent = "unknown". This is a valid KIF
      simplification (Bonet & Geffner style), but it deviates from the
      proposal's two-predicate design without documenting the change.

PA-4. Sensing uses fixed camera instead of robot-mounted sensor
      The proposal's sense action uses stream_find_sensing_config to
      compute a sensing pose. The implementation uses a fixed scene
      camera. This is an accepted simplification for the tabletop
      scenario — a fixed overhead camera is sufficient and avoids
      the complexity of a separate sensing motion planner. Should be
      documented as a deliberate design decision in the thesis.

PA-5. No conditional sensing outcomes (proposal Section 4.4.2)
      The proposal defines conditional effects for sense: (when found ...)
      and (when not_found ...) using a stream_get_sensing_outcome stream.
      The implementation uses optimistic single-outcome sensing with
      reactive replanning on failure.  This prevents the planner from
      generating multi-step search strategies.  See #61 for full details.

PA-6. Only top-down grasps — no grasp diversity
      sample_grasp() generates only top-down grasps (Z-axis offsets at
      0.05/0.10/0.15 m, fixed orientation pointing down).  No lateral
      or angled grasps are considered.  While the proposal doesn't
      explicitly specify grasp diversity, the framework's ability to
      handle complex scenes (objects at walls, stacked objects, narrow
      gaps) is limited without it.  This restricts the evaluation
      scenarios to simple tabletop layouts where top-down access is
      always possible.

PDDL ACTION / PYTHON FUNCTION ALIGNMENT
================================================================================
Cross-reference of every PDDL action against its Python execution code.
See proposal Section 4.4 for the intended mapping.

PF-1. sense — PDDL vs Python MISMATCH
      PDDL (domain lines 120-133): Two params (?o ?region).  Precondition:
        view_clear. Effect: unconditionally adds obj_at_boxel, obj_pose_known.
      Python (test_full_pipeline.py lines 303-337): Calls
        sense_shadow_raycasting() which returns one of three outcomes
        (found_target, clear_but_empty, still_blocked).  Only "found_target"
        updates belief as success.  The other two break to replan.
      Mismatch: PDDL has no stream for observation outcome and no
        conditional effects.  It cannot represent "not found" or "still
        blocked" — those are only handled by the reactive replan loop.
        See #61.

PF-2. pick — PDDL vs Python MISMATCH (scope)
      PDDL (domain lines 161-179): Single config ?q.  Instantaneous.
        Does not update at_config.
      Python (execute_pick, lines 492-560): 4-step sequence (open gripper,
        approach, contact, close, attach constraint, lift).  Re-derives
        contact IK from actual obj_pos at execution time.  Leaves robot
        at lift config, not pick config.
      Mismatch: Python does MORE than PDDL describes (re-solve IK, create
        constraint, multi-waypoint motion).  PDDL model does not capture
        the post-pick config drift.  See #62.

PF-3. place — PDDL vs Python MISMATCH (scope)
      Same pattern as PF-2.  PDDL models single-config instantaneous action.
      Python executes approach → contact → release → settle → retreat.
      Post-place config differs from PDDL's ?q.  See #62.

PF-4. move — CONSISTENT
      PDDL (domain lines 138-155): Moves from ?q1 to ?q2 along trajectory ?t.
        Updates at_config.
      Python (test_full_pipeline.py lines 293-300): Iterates through
        traj.waypoints[1:], calling move_robot_smooth() per waypoint.
      Status: Consistent.  Trajectory is faithfully executed.

PF-5. compute-kin stream — HIDDEN SIDE-CHANNEL
      Stream PDDL (stream.pddl lines 35-40): Inputs (?o ?b ?g), outputs ?q.
        Certifies Config, kin_solution, config_for_boxel.
      Python (streams.py compute_kin_solution, lines 770-854): Yields
        RobotConfig with ignored_body_ids (frozenset of body IDs to exclude
        from collision checks).
      Issue: ignored_body_ids is not visible in the PDDL declaration.
        plan_motion() reads it from the Python RobotConfig object and uses
        it to exclude the grasped object.  The symbolic planner has no
        knowledge of this collision exclusion logic — it's a hidden Python
        side-channel.  This is architecturally sound for PDDLStream (streams
        share Python objects by design) but invisible to formal analysis.
