44. SHADOWS NOT RECOMPUTED AFTER OCCLUDER RELOCATION — STALE REGISTRY
================================================================================
Where: test_full_pipeline.py, boxel_data.py
What:  After relocating an occluder (via pick-and-place), the BoxelRegistry
       is NOT updated. The shadow boxels (their AABBs, centers, extents)
       still reflect the pre-relocation geometry. The sensing function
       (sense_shadow_raycasting) uses the shadow boxel's AABB to determine
       where to cast rays — but that AABB is for the old shadow, not the
       actual post-relocation geometry.

       With pick-and-place this is LESS CRITICAL than with push: since the
       occluder is physically removed (picked up and placed elsewhere),
       rays through the old shadow AABB will pass through empty space and
       can detect the target. The stale AABB just means rays may cover a
       slightly wrong area, but the target is detected by body ID, so any
       ray that hits it works.

Fix:   After relocating, either:
       (a) Recompute shadows for the relocated occluder only (incremental).
       (b) Re-run the full boxel computation (expensive but correct).
       (c) Accept the stale AABB as good enough (target detection by body
           ID compensates). Document this as a known simplification.

59. OBJECTS FLY AWAY ON FIRST GRASP — CONSTRAINT-BASED GRASPING IS THE PROBLEM
================================================================================
Where: test_full_pipeline.py execute_pick() / execute_place(),
       robot_utils.py close_gripper()

What:  On the first pick action, surrounding objects visibly fly away from
       the grasped object. The grasp itself also looks "supernatural" — the
       object snaps to the gripper rather than being smoothly acquired.

       ROOT CAUSE: execute_pick() uses p.createConstraint(JOINT_FIXED) to
       "weld" the object to the end-effector.  This is not grasping — it is
       a telekinetic attachment that bypasses contact physics entirely.  The
       constraint solver fights the rigid-body simulator every stepSimulation
       to maintain the weld, producing large corrective impulses that
       propagate to adjacent objects via contact forces.

       This is not a tuning problem (wrong offsets, missing orientations,
       missing maxForce).  Even a perfectly parameterised JOINT_FIXED
       constraint is still a magic weld: it can hold objects that don't fit
       in the gripper, it ignores friction, it ignores gravity on the
       grasped body, and it allows the gripper to remain open while the
       object stays attached.  The snap-to-gripper visual artefact and the
       neighbour-scattering impulse are inherent to this approach.

Fix:   Replace constraint-based grasping with FRICTION-BASED grasping:
       1. Remove p.createConstraint / p.removeConstraint from execute_pick
          and execute_place.
       2. Rely on close_gripper() finger contact + friction to hold the
          object.  The gripper motors already apply 50 N per finger; with
          a friction coefficient ≥ 0.5 the resulting friction force far
          exceeds the weight of any scene object.
       3. open_gripper() releases the object (no constraint to remove).
       4. Adjust grasp Z offsets in sample_grasp() so the EE descends to
          the object's center height (not above it).  The current offsets
          (0.05–0.15 m above object center) place the fingers in the air
          above the object — fine for a magic weld, useless for friction.
       5. Set lateralFriction on object and gripper collision shapes to
          ensure reliable grip.

       Depends on #77 (object graspability): current object dimensions
       exceed or match the Panda's 0.08 m max finger opening, so friction-
       based grasping cannot work without resizing.  See #77.

60. SIMULATION-WIDE FLICKERING DURING PLANNING — RENDERING TOGGLE ARTIFACTS  [DONE]
================================================================================
Where: robot_utils.py is_config_collision_free(), is_path_collision_free(),
       solve_ik(); streams.py _pybullet_ik()

What:  After the arm-flickering fix (wrapping resetJointState with
       configureDebugVisualizer COV_ENABLE_RENDERING 0/1), the arm no
       longer flickers, but the entire simulation now exhibits periodic
       visual glitches — the scene briefly looks "weird" every couple of
       seconds.

       Root cause: the rendering toggle is called thousands of times per
       planning cycle (once per is_path_collision_free call, once per
       solve_ik/IK call).  Each enable→disable→enable cycle may force
       an OpenGL buffer swap or partial frame render, producing a brief
       flash.  The fix addressed per-call arm teleportation but introduced
       a high-frequency global rendering toggle that has its own visual
       side effects.

Fix:   IMPLEMENTED — reference-counted RenderingLock (robot_utils.py).
       The first acquire (0→1) disables rendering; the last release (1→0)
       re-enables it.  All four toggle sites (is_config_collision_free,
       is_path_collision_free, solve_ik, _pybullet_ik) now use
       RenderingLock instead of raw configureDebugVisualizer calls.
       test_full_pipeline.py wraps planner.plan() with an outer
       RenderingLock — the inner locks nest harmlessly via the reference
       count, so rendering is toggled exactly TWICE per planning phase
       (once off, once on) regardless of how many IK/collision calls occur.
       Standalone calls outside planning still work: the lock auto-toggles
       when the counter transitions 0→1 or 1→0.

61. SENSE ACTION DEVIATES FROM PROPOSAL — NO CONDITIONAL EFFECTS  [ACCEPTED]
================================================================================
Where: pddl/domain_pddlstream.pddl (sense action),
       test_full_pipeline.py (sense execution + replan loop),
       proposal Section 4.4.2 (Listing 2)

What:  The PDDL sense action unconditionally adds (obj_at_boxel ?o ?region)
       and (obj_pose_known ?o).  It assumes the target will always be found
       ("optimistic").  The proposal's conditional design with
       stream_get_sensing_outcome and (when found/not_found ...) branches
       is not implemented.  Instead, the execution loop detects sensing
       failures and replans with updated belief (known_empty_shadows).

Status: ACCEPTED DEVIATION — documented in PDDL domain file and code.

Justification:
       (a) PDDLStream + FastDownward cannot do contingent planning.
           Branching on observation outcomes requires a contingent planner
           (POND, CLG), which is outside PDDLStream's architecture.  The
           proposal's conditional-effect design cannot be implemented
           within the chosen planning framework without fundamental changes.
       (b) Optimistic planning with replanning on failure is a standard
           TAMP pattern under partial observability (Garrett et al., 2020;
           Kaelbling & Lozano-Pérez, 2013).  PDDLStream's adaptive
           algorithm is itself built around optimistic assumptions.
       (c) For N shadow candidates, reactive replanning converges in at
           most N replan cycles (one per empty shadow).  Belief state
           propagates correctly: known_empty_shadows carry over, so each
           replan searches strictly fewer candidates.
       (d) The execution loop bounds the search: max_replans = 4*N + 1.

Limitation:
       The planner cannot optimize search ORDER within a single plan.
       A contingent planner could reason about most-likely-first strategies.
       For the current uniform-prior tabletop scenario this does not affect
       completeness or practical efficiency.

67. _build_init() HARDCODES ASSUMPTION THAT TARGETS ARE ONLY IN SHADOWS  [DONE]
================================================================================
Where: pddlstream_planner.py _build_init(), lines 269-286

What:  For every OBJECT boxel (line 280-281) and every FREE_SPACE boxel
       (lines 285-286), the code adds:
         (obj_at_boxel_KIF target boxel_id)
       This means "we know the target is NOT in this boxel" — because
       obj_at_boxel is not also added.  Only shadow boxels are left as
       "unknown" (no KIF fact emitted unless in known_empty_shadows).

       This hardcodes the assumption that targets can ONLY be hidden in
       shadow regions.  The proposal's formulation is more general — a
       target could be anywhere the robot hasn't looked.  The current
       implementation precludes scenarios where a target is on an open
       part of the table but the robot doesn't know which one it is.

Fix:   DONE — _build_init() now takes an optional observed_clear_regions
       parameter.  When None (default), all object/free-space boxels are
       treated as observed (preserving current behavior for the fixed
       overhead camera scenario).  When an explicit set is provided, only
       those regions get KIF facts.  The assumption is documented in the
       docstring with guidance for extending to partial-coverage sensors.

Impact: Low for the current hidden-object scenario (targets are always
       behind occluders by construction).  But this prevents extending
       the system to more general search tasks without reworking _build_init.

PROPOSAL ALIGNMENT GAPS
================================================================================
These are features described in the proposal that have no corresponding
implementation. They are not code bugs — they are missing functionality
that the thesis promises.

PA-1. No evaluation framework (proposal Section 5)
      The proposal defines metrics (success rate, planning time, plan cost,
      scalability) and three baselines (uniform voxelization, fixed semantic
      regions, POMDP-based TAMP). None exist in the codebase. There is no
      benchmarking code, no baseline implementations, no metric collection,
      no experiment runner.

PA-2. No recursive object discovery in free space (proposal Section 4.2)
      The proposal says: "If new objects are found within a partition, the
      process repeats: bound the biggest object and define its occluded
      space." The actual FreeSpaceGenerator does a single-pass octree that
      marks cells as FREE or OCCUPIED. It never detects new objects inside
      partitions — all objects are detected upfront by oracle_detect_objects.

PA-3. K-literal design differs from proposal (proposal Section 4.4.1)
      The proposal defines two predicates: obj_in_Boxel (K(p)) and
      obj_not_in_Boxel (K(¬p)), with "possibly in" being the absence of
      both. The implementation uses a single obj_at_boxel_KIF predicate:
      present = "we know", absent = "unknown". This is a valid KIF
      simplification (Bonet & Geffner style), but it deviates from the
      proposal's two-predicate design without documenting the change.

PA-4. Sensing uses fixed camera instead of robot-mounted sensor
      The proposal's sense action uses stream_find_sensing_config to
      compute a sensing pose. The implementation uses a fixed scene
      camera. This is an accepted simplification for the tabletop
      scenario — a fixed overhead camera is sufficient and avoids
      the complexity of a separate sensing motion planner. Should be
      documented as a deliberate design decision in the thesis.

PA-5. No conditional sensing outcomes (proposal Section 4.4.2) — ACCEPTED
      The proposal defines conditional effects for sense: (when found ...)
      and (when not_found ...) using a stream_get_sensing_outcome stream.
      The implementation uses optimistic single-outcome sensing with
      reactive replanning on failure.  Accepted deviation: PDDLStream +
      FastDownward cannot do contingent planning; optimistic planning with
      replanning is a standard TAMP pattern.  See #61 for full justification.

PA-6. Only top-down grasps — no grasp diversity
      sample_grasp() generates only top-down grasps (Z-axis offsets at
      0.05/0.10/0.15 m, fixed orientation pointing down).  No lateral
      or angled grasps are considered.  While the proposal doesn't
      explicitly specify grasp diversity, the framework's ability to
      handle complex scenes (objects at walls, stacked objects, narrow
      gaps) is limited without it.  This restricts the evaluation
      scenarios to simple tabletop layouts where top-down access is
      always possible.

PDDL ACTION / PYTHON FUNCTION ALIGNMENT
================================================================================
Cross-reference of every PDDL action against its Python execution code.
See proposal Section 4.4 for the intended mapping.

PF-1. sense — PDDL vs Python MISMATCH (accepted, see #61)
      PDDL: optimistic single-outcome (assumes found).
      Python: tri-state (found / clear_but_empty / still_blocked) with
        reactive replanning on failure.
      Accepted deviation: contingent planning not available in PDDLStream;
        optimistic sensing + replanning is functionally equivalent for the
        tabletop scenario.  Documented in domain PDDL and code.

PF-2. pick — PDDL vs Python MISMATCH (scope)
      PDDL (domain lines 161-179): Single config ?q.  Instantaneous.
        Does not update at_config.
      Python (execute_pick, lines 492-560): 4-step sequence (open gripper,
        approach, contact, close, attach constraint, lift).  Re-derives
        contact IK from actual obj_pos at execution time.  Leaves robot
        at lift config, not pick config.
      Mismatch: Python does MORE than PDDL describes (re-solve IK, create
        constraint, multi-waypoint motion).  PDDL model does not capture
        the post-pick config drift.  See #62.

PF-3. place — PDDL vs Python MISMATCH (scope)
      Same pattern as PF-2.  PDDL models single-config instantaneous action.
      Python executes approach → contact → release → settle → retreat.
      Post-place config differs from PDDL's ?q.  See #62.

PF-4. move — CONSISTENT
      PDDL (domain lines 138-155): Moves from ?q1 to ?q2 along trajectory ?t.
        Updates at_config.
      Python (test_full_pipeline.py lines 293-300): Iterates through
        traj.waypoints[1:], calling move_robot_smooth() per waypoint.
      Status: Consistent.  Trajectory is faithfully executed.

PF-5. compute-kin stream — HIDDEN SIDE-CHANNEL
      Stream PDDL (stream.pddl lines 35-40): Inputs (?o ?b ?g), outputs ?q.
        Certifies Config, kin_solution, config_for_boxel.
      Python (streams.py compute_kin_solution, lines 770-854): Yields
        RobotConfig with ignored_body_ids (frozenset of body IDs to exclude
        from collision checks).
      Issue: ignored_body_ids is not visible in the PDDL declaration.
        plan_motion() reads it from the Python RobotConfig object and uses
        it to exclude the grasped object.  The symbolic planner has no
        knowledge of this collision exclusion logic — it's a hidden Python
        side-channel.  This is architecturally sound for PDDLStream (streams
        share Python objects by design) but invisible to formal analysis.

73. shadow_occluder_map GOES STALE ON REPLAN — blocks_view_at MAY BE WRONG
================================================================================
Where: test_full_pipeline.py lines 185-189,
       pddlstream_planner.py _build_init() lines 292-300

What:  shadow_occluder_map is built once from the registry before the
       first plan.  After an occluder is relocated via pick-and-place,
       the map is NOT updated.  On subsequent replanning iterations,
       _build_init() still emits:
         (blocks_view_at old_occ old_occ shadow_id)
       This says "when old_occ is at its original boxel, it blocks the
       view to shadow_id".  The derived predicate blocks_view combines
       this with obj_at_boxel — if the occluder was placed in free
       space, obj_at_boxel(old_occ, old_occ) is false, so blocks_view
       evaluates to false and the view is correctly classified as clear.

       HOWEVER, if the planner generated a new blocks_view_at fact
       that references the destination boxel (it doesn't — but if it
       did, or if the PDDL model ever added view-blocking at the new
       location), the stale map would miss it.

       Current impact: LOW because the derived predicates compensate.
       But the stale data structure is architecturally fragile and
       should be documented or refreshed on replan.

77. OBJECT DIMENSIONS EXCEED GRIPPER CAPACITY — FRICTION GRASPING BLOCKED
================================================================================
Where: boxel_env.py _create_occluders(), _create_targets()

What:  The Franka Panda parallel-jaw gripper has a maximum finger opening
       of 0.08 m (0.04 m per finger).  For friction-based grasping the
       object's graspable width must be strictly LESS than the max opening,
       so the fingers can close around the object and apply normal force.

       Current scene objects:
         Occluders  0.15 m cubes  — 1.9× the max opening.  Physically
                                    ungraspable by the Panda gripper.
                                    The comment on boxel_env.py line 178
                                    ("small enough for the Panda gripper
                                    to grasp") is INCORRECT.
         Targets    0.08 m cubes  — exactly AT the max opening.  Fingers
                                    can barely touch the sides but have
                                    zero squeeze margin; unreliable grip.

       With the current constraint-based "grasping" (#59) this didn't
       matter — the constraint can hold any object regardless of size.
       Switching to friction-based grasping (the #59 fix) requires objects
       that physically fit in the gripper.

Fix:   Resize objects so their graspable width is < 0.08 m:
       (a) Targets: reduce to ~0.05 m cubes.  Still hidden behind
           occluders, easier to grasp with 0.03 m of finger margin.
       (b) Occluders: change from cubes to tall narrow blocks
           (e.g. 0.06 × 0.06 × 0.15 m).  The 0.06 m width fits in the
           gripper; the 0.15 m height still occludes targets from the
           overhead camera.  Alternatively, occluders could be cylinders
           (diameter < 0.08 m) for even easier top-down grasping.
       (c) Set lateralFriction on object and gripper collision shapes
           (PyBullet default for createCollisionShape is 0.5; verify
           it is sufficient, increase to ~1.0 if needed).

Impact: LOW for the architecture — only object spawn parameters change.
       HIGH for #59 — friction-based grasping cannot work until this
       is addressed.

78. INCOMPLETE VIEW-BLOCKING MODEL — PIPELINE LOOPS FOREVER ON SOME SHADOWS  [DONE]
================================================================================
Where: pddlstream_planner.py _build_init() lines 291-303,
       test_full_pipeline.py lines 181-188 (shadow_occluder_map),
       boxel_env.py generate_boxels()

What:  The blocks_view_at facts in the planner init state only model
       the CREATING occluder for each shadow (from shadow_occluder_map).
       But the camera's line of sight to a shadow can be blocked by ANY
       object in the scene — not just the one that geometrically cast
       the shadow.

       Observed failure (run_2026-03-14_16-10-02):
         - shadow_001 is created by occluder_3 (obj_004).
         - shadow_occluder_map records: shadow_001 → obj_004.
         - Plan #3 correctly picks up occluder_3 and places it in free space.
         - But occluder_1 (at [0.5, 0.2, z]) physically sits between the
           camera and shadow_001. It blocks 55% of sensing rays.
         - The planner's view_clear(shadow_001) derived predicate evaluates
           to TRUE because obj_004 was moved — but occluder_1 was never
           modeled in blocks_view_at for shadow_001.
         - Plans #4-#10 all generate move → sense → pick, sensing keeps
           reporting "still_blocked (55%)", and the loop repeats until
           the replan limit is hit.

       Two sub-problems:
       (a) blocks_view_at only records the creating occluder, not all
           objects that physically obstruct the camera → shadow corridor.
       (b) occluder_1 is visible but does NOT appear in the boxel registry
           as an OBJECT boxel (its shadow falls off the table or is too
           small). Without an OBJECT boxel, the planner cannot plan to
           move it. Even if blocks_view_at included it, there would be
           no (Obj occluder_1) or (obj_at_boxel occluder_1 ...) facts.

Fix:   DONE — (a) compute_shadow_blockers() in test_full_pipeline.py
       uses camera raycasting to find ALL objects blocking each shadow.
       shadow_occluder_map is now Dict[shadow_id, List[blocker_ids]].
       _build_init() in pddlstream_planner.py emits blocks_view_at for
       every (blocker, shadow) pair.
       (b) Verified: generate_boxels() → create_boxel_registry_from_boxels
       preserves all visible objects as OBJECT boxels regardless of
       whether they cast shadows.
       (c) Execution loop tracks blocked_counts per shadow; after 3
       consecutive "still_blocked" outcomes on the same shadow, marks
       it as empty and moves on.

Impact: HIGH — causes the pipeline to loop forever on scenes where a
       non-creating object blocks a shadow's view corridor. This is not
       a corner case: in the current 3-occluder scene, it happens
       whenever the target is behind the occluder whose shadow is also
       blocked by another occluder from the camera's perspective.

93. ACTIONS ARE NOT GENERIC — HIDDEN SUB-ACTIONS EMBEDDED IN EXECUTION
================================================================================
Where: test_full_pipeline.py (sense handler, execute_pick, execute_place)

What:  PDDL actions are modeled as atomic operations, but their Python
       execution handlers secretly perform operations that belong to
       OTHER action types.  The planner cannot reason about, plan for,
       or optimize these hidden sub-actions.

       sense handler (lines 340-345):
         Contains a hidden MOVE: moves the arm to home configuration
         before raycasting.  The planner doesn't know sensing requires
         the arm to be out of the way — it has no (at_config ?q_home)
         precondition on the sense action.  This was added as a
         workaround for #79 (arm blocking camera), but embedding a
         move inside sense violates action atomicity.

       execute_pick (lines 630-714):
         Contains 3 hidden MOVEs (approach, descend to contact, lift),
         3 hidden IK SOLVES (re-deriving geometry the planner never
         planned), a GRIPPER_OPEN, a GRIPPER_CLOSE, and a CONSTRAINT
         WELD (p.createConstraint).  The PDDL pick action is a single
         instantaneous operation; the Python handler is a 6-step
         choreographed sequence.

       execute_place (lines 717-789):
         Contains 3 hidden MOVEs (approach, descend, retreat), 3 hidden
         IK SOLVES, a GRIPPER_OPEN, a CONSTRAINT REMOVAL, and 30 hidden
         PHYSICS STEPS for object settling.  Same atomicity violation
         as pick.

       Complete inventory of hidden sub-actions:
         sense:  1 move
         pick:   3 moves, 3 IK solves, 1 open_gripper, 1 close_gripper,
                 1 createConstraint
         place:  3 moves, 3 IK solves, 1 open_gripper,
                 1 removeConstraint, 30 stepSimulation

       Consequences:
       (a) The planner cannot optimize the approach/retreat motions or
           detect collisions on them — they are invisible to the
           symbolic search.
       (b) The execution-time IK solves (solve_ik in execute_pick/place)
           can fail, producing None.  These failures are handled by
           abort-and-replan (#82), but the planner has no awareness
           that its "valid" plan might fail at execution.
       (c) The hidden move in sense causes PDDL state drift — the robot
           moves to home but the PDDL state still says (at_config ?q)
           from the last move action.  Compensated by current_config
           tracking (#86) but architecturally unsound.
       (d) The approach/retreat heights (0.10 m, 0.25 m) are hardcoded
           constants that the planner never validates for collisions.

       Related audit entries:
         #79  Arm blocks sensing — the hidden move was the workaround
         #82  Silent IK failures — hidden IK solves can fail
         #86  PDDL state drift — hidden moves cause drift
         PF-1 sense PDDL/Python mismatch (accepted #61)
         PF-2 pick PDDL/Python mismatch (documented)
         PF-3 place PDDL/Python mismatch (documented)

Fix:   To make actions truly generic, decompose each into sub-actions
       in the PDDL domain:
       (a) sense: add (at_config ?q_home) precondition, or add a
           separate prepare_sense action that moves the arm home.
       (b) pick: decompose into approach(?o ?b ?g ?q_approach),
           grasp(?o), lift(?o ?q_lift) — each with their own
           compute-kin stream calls and plan-motion trajectories.
       (c) place: decompose into approach(?o ?b ?g ?q_approach),
           release(?o), retreat(?q_retreat).
       Cost: 3x more stream evaluations per pick/place, longer plans,
       and more complex domain.  The current hardcoded approach works
       functionally and is a common simplification in TAMP research.

95. SYSTEMIC PDDL STATE DRIFT — SYMBOLIC/PHYSICAL DESYNC ACROSS ACTIONS
================================================================================
Where: domain_pddlstream.pddl (all actions),
       test_full_pipeline.py (sense handler, execute_pick, execute_place),
       pddlstream_planner.py (_build_init),
       boxel_data.py (BoxelRegistry)

What:  The PDDL symbolic state and the PyBullet physical state diverge
       systematically during plan execution.  This is not a single bug —
       it is a structural pattern affecting every action that modifies
       the world.  Four distinct drift channels have been identified:

       (a) ROBOT CONFIG DRIFT (pick/place) — #86, PF-2, PF-3
           PDDL pick/place are modeled as instantaneous single-config
           actions.  Python execution performs multi-waypoint sequences
           (approach → contact → lift/retreat) and leaves the robot at a
           post-action configuration that PDDL never learns about.  After
           pick, PDDL says (at_config ?q_pick) while the robot is at
           post_pick_lift.  After place, PDDL says (at_config ?q_place)
           while the robot is at post_place_retreat.
           Mitigation: execution reads actual p.getJointState and passes
           it to the planner on replan (#86 DONE).  Drift still exists
           WITHIN a single plan — if the plan chains pick → move, the
           move starts from the wrong config in the symbolic model.

       (b) HIDDEN MOVE IN SENSE — #93, #79
           The sense handler secretly moves the robot to home before
           raycasting (workaround for #79).  PDDL has no at_config
           precondition or effect on sense — the symbolic state still
           records (at_config ?q) from the previous move action.
           Mitigation: current_config updated in execution loop.  Same
           within-plan drift as (a).

       (c) STALE SHADOW GEOMETRY — #44
           After an occluder is relocated via pick-and-place, the
           BoxelRegistry is NOT recomputed.  Shadow boxel AABBs still
           describe the pre-relocation geometry.  Sensing uses these
           stale AABBs to target raycasts.
           Mitigation: target detection is by PyBullet body ID, so any
           ray that hits the target works regardless of AABB accuracy.
           Fragile if the scene changes more substantially.

       (d) STALE BLOCKS_VIEW_AT FACTS — #73
           shadow_occluder_map is built once before the first plan.
           After occluder relocation, _build_init() still emits
           blocks_view_at facts for the original occluder positions.
           Mitigation: derived predicate blocks_view checks
           obj_at_boxel, which IS updated — so the stale facts are
           functionally dead.  But the data structure is fragile:
           any extension that references blocks_view_at directly
           (rather than through the derived predicate) would break.

       Systemic consequences:
       - Within a single plan, the symbolic and physical states are
         out of sync after every non-move action.  If the planner
         generates plans with pick → move or sense → pick sequences,
         the move/pick operates on a config the robot is NOT at.
       - The execution-side mitigations (#86 current_config tracking)
         only take effect on REPLAN — they cannot fix drift within
         a single plan's action sequence.
       - The stale geometry data (#44, #73) accumulates across replans
         because the registry is never refreshed.

       This is the root architectural issue underlying #44, #73, #86,
       and #93.  Fixing it properly requires either:
       (a) Making the PDDL model faithful — decompose pick/place into
           sub-actions, add at_config to sense, update registry on
           replan.  This is the #93 fix plus registry refresh.
       (b) Adding execution-time state reconciliation — before each
           action in a plan, re-derive configs from the actual physical
           state rather than trusting the plan's symbolic parameters.
       (c) Both: faithful PDDL model + execution-time validation.

Related: #44, #73, #86 (DONE), #93, PF-2, PF-3

       Commented-out code (e.g. visualization calls, disabled features,
       archived logic) is preserved intentionally.  It serves as inline
       documentation of previous approaches, quick-enable toggles for
       debugging, and reference for future re-implementation.

       - test_full_pipeline.py line 146: draw_boxels call (disabled to
         simplify pipeline runs; re-enable for visual debugging)
       - archive/ folder: entire previous implementations kept for
         reference

       Rule: NEVER delete commented-out code without explicit approval.
       If code is truly obsolete, move it to archive/ rather than
       deleting it.

       Dead methods, unused imports, and unreachable code paths documented
       in issues #87–#92 are tracked for awareness but are NOT scheduled
       for deletion.  They impose no runtime cost and may be needed for
       future features (evaluation framework, real perception, etc.).

WIKI — UPDATE AFTER EVERY CHANGE
       The project wiki lives in the GitHub wiki repo (pybullet.wiki.git),
       NOT in the main repository.  After any code change that affects
       architecture, data structures, algorithms, PDDL domain, execution
       pipeline, or design decisions, the corresponding wiki page(s) MUST
       be updated to stay in sync.

       Wiki repo: git@github.com:hanihabobo-dot/pybullet.wiki.git
       Local clone: ../pybullet.wiki/

         Architecture_Overview     — new/removed modules, changed data flow

         Planning_System           — _build_init, streams, replanning
         Robot_Control_and_Streams — IK, collision, motion planning, grasps
         Execution_Pipeline        — action handlers, sensing, replanning
         PDDL_Domain_Reference    — predicates, actions, streams
         Design_Decisions          — new accepted deviations or rationale
         Known_Issues_and_Roadmap  — resolved/new audit issues

       The archive/ folder contains superseded implementations preserved
       for local reference.  It MUST NOT contain personal information

       added to .gitignore so it is not pushed to the remote repository.
       If it is currently tracked, it should be removed from tracking
       with `git rm -r --cached archive/` and then ignored.
